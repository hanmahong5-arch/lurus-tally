## Story

As founder + dev team,
I want a Monday 09:00 飞书 weekly health report 显示 features_shipped / bugs_open / assumption_evidence_gap_days / lint_warnings_total / feature-to-WAD ratio,
so that "Anti-metric 升级" 段不沦为"挂在 doc 没人看"的目标，团队周一 retro 直接拿数说话。

## Acceptance Criteria

1. 新建 `bin/health-report.sh`（POSIX bash，shellcheck 绿），输出**单条飞书消息**到 `HEALTH_REPORT_FEISHU_URL` webhook，markdown 格式如下：
   ```
   📊 Tally V1.5 Health · Week <YYYY-Www>

   • features_shipped (last 7d): N (target ≤ 3/月 ≈ 0.7/周)
   • bugs_open - bugs_closed delta: ±N (target net ≤ 5 同时 open)
   • assumption_evidence_gap_days: H1=X, H2=Y, H3=Z (target ≤ 7d)
   • lint_warnings_total: N (last week: M, change: ±%) ← 必须单调递减
   • features_shipped_cumulative / WAD_weekly_avg: 0.XX (target 必降, 当前: 上升/平/降)

   红线：<任一 anti-metric 连 2 周违 → @founder retrospective>
   ```
2. 数据来源：
   - `features_shipped`: count GH issue closed with label `feature` in last 7d via `gh issue list --label feature --state closed --search 'closed:>=...'`
   - `bugs_open - bugs_closed`: 同上 label `regression` 或 `bug`
   - `assumption_evidence_gap_days`: 读 `_bmad-output/planning-artifacts/assumptions.md` 末尾历史表，找最后非 n/a 行，今日 - 该日期
   - `lint_warnings_total`: 从最近 main 的 ci.yaml 跑 artifact 拉 `lint-baseline.txt` 计数；如无 artifact → 跳过当周（写 `data n/a`）
   - `WAD_weekly_avg`: Prom query `sum(rate(tally_plan_accept_total[7d])) * 86400 * 7`
3. cron schedule `0 1 * * 1`（北京时间周一 09:00），落 governance repo `.github/workflows/health-report.yaml` 推荐方案。
4. 失败时（webhook 504 / curl 非 0 / Prom 不可达）发到 backup webhook `OPS_ALERT_FEISHU`，不静默吞。
5. **本 sprint 跑过 1 次**：手动 `bash bin/health-report.sh` 触发；飞书群里见格式正确的消息（即使数据多 n/a）。截图贴到 Dev Agent Record。
6. 不能在 health-report 里塞预测 / 建议 — 只报数；解读由周一 retro 人决定。

## Dev Notes

- **不要 over-engineer** — 本 story 就是周一拉数发消息；不引入数据库 / cron monitoring 平台。
- 头 4 周大概率 N/A 满天飞，OK。**周报"准时发"比"数据全"重要** — 这正是 Anti-metric 升级的目的：让"无数据"本身成为信号。
- 如 founder 想加新指标（如 NPS / churn），先在 retro 提议、改 roadmap-v1.5 Anti-metric 表、再改本脚本。**不在 script 自由扩字段**。
- WAD 定义见 roadmap-v1.5 North Star 段；目前 metric 名 `tally_plan_accept_total` 是 Sprint 5 才上线的，前 5 sprint 报"n/a"。
- 飞书 markdown 不支持 table，所以输出用 bullet list 形式。
- 不要在脚本里调 OpenAI / 任何 LLM 做"自动 retrospective" — 那是反原则一"AI 不是卖点"。
