## 1. Product Vision

### 1.1 一句话定位

**Lurus Tally 是中国中小企业的第一款 AI-native 进销存 SaaS —— 让补货决策、库存分析和合规检查都由 Agent 自动完成，老板只需审核和拍板。**

### 1.2 为什么是现在

国内 PSI 市场正在经历两个并行压力点，二者叠加创造出一个 **12-18 个月的稀缺窗口期**（见 google.md §1）：

1. **金税四期全面落地**（2025-2026）——不支持数电票 XML 归档和四流一致检查的旧系统正在被客户主动淘汰。用友/金蝶已在大中型客户侧完成布局，SMB 侧空缺仍大（见 google.md §5.6）。
2. **AI-native PSI 真空**——国内目前无产品实现真正的"自主补货决策 Agent"闭环。简道云最接近但它是通用低代码平台，无垂直深度；传统进销存（管家婆/速达）AI 能力浮于表面（见 google.md §3）。

Lurus 凭借已有的 Kova Agent 引擎 + Hub LLM 网关，具备构建 AI-native PSI 的全部技术底座，新进入者需要至少 12-18 个月才能追上。

### 1.3 六个月成功指标

| 指标 | 目标值 | 备注 |
|------|-------|------|
| Lighthouse 客户数 | 3+ | M3 内部 dogfood 阶段 |
| Stage 付费客户 | 20+ | M6 MVP β 阶段 |
| AI 助手日活查询 | ≥ 每用户 2 次/天 | ⌘K + Drawer 触发总数 |
| 补货建议采纳率 | ≥ 40% | Kova Agent 建议被用户执行的比例 |
| 客户月留存率 | ≥ 85% | 付费续费留存 |
| 平均首次开单时间 | < 10 分钟 | 新用户注册到完成第一张销售单 |

## 2. Strategic Context

### 2.1 与 Lurus 既有体系的关系

Tally 不是孤立产品，它是 Lurus Platform 产品组的第五个成员，深度复用已有基础设施：

| Lurus 能力 | Tally 使用方式 | 节省建设成本 |
|-----------|--------------|-------------|
| **2l-svc-platform** | 账户/钱包/订阅/权益，调 `/internal/v1/...` (bearer key) | 6-12 个月 |
| **2b-svc-api (Hub)** | 所有 LLM 调用走 Hub 网关，Hub 负责路由/计量/熔断 | 3-6 个月 |
| **2b-svc-kova** | Kova Agent 引擎执行补货分析、滞销预警等 Agent 任务 | 6-9 个月 |
| **2b-svc-memorus** | RAG 历史销量/客户偏好上下文，提升 AI 预测精度 | 3-6 个月 |
| **Zitadel (OIDC)** | 单点登录，复用 auth.lurus.cn | 2-3 个月 |
| **PostgreSQL lurus-pg-rw** | 共用集群，`tally` schema，Row-Level Security 多租户 | 1-2 个月 |
| **NATS** | 新增 `PSI_EVENTS` stream，与 IDENTITY_EVENTS/LLM_EVENTS 并列 | 1 个月 |

**产品组归属**: Platform (P0)。Tally 的订阅收入、AI 调用量、客户数据均纳入 Platform 产品线统一管理。

### 2.2 与竞品的核心差距

| 竞品 | 他们的护城河 | Tally 的优势 |
|------|------------|------------|
| **管家婆** | 本地化服务网络、断网开单 | AI Agent 补货决策、⌘K 自然语言查询、顶级 UX |
| **速达** | 制造业生产模块覆盖 | Web-first 更快上手、AI 异常检测、金税四期深度集成 |
| **简道云 PSI** | 零代码自定义、AI 动态定价 | 垂直深度（批次/序列号/多仓）、Kova Agent（非规则引擎）|
| **有赞** | 私域电商生态、微信 | 进销存数据深度、AI 补货非营销导向 |
| **Cin7** | 100+ 渠道集成、ForesightAI | 中文语境、金税四期合规、Lurus Hub 计费体系 |

Tally 不打算正面竞争任何一家，战略切入点是：**AI Agent + 金税四期合规 + Linear 级 UX** 三者同时具备，这在国内是空白组合。

## 3. Target Users — Personas

### Persona A: 老板娘 李青 (决策者 + 主要付费人)

**背景**: 经营一家服装批发店，年营收约 600 万，5 名员工，抖店 + 实体批发双渠道运营。
**现状**: 用管家婆 7 年，每月月底要手动 Excel 对账，补货决策完全凭经验，经常超卖或积压。
**核心痛点**（来自 google.md §6）:
- 多平台库存手动维护，抖店和批发客户经常抢同一批货导致超卖
- 补货决策依赖老员工经验，旺季来不及反应
- 月底财务对账要花 2-3 天，和会计反复核单

**Tally 解决的核心场景**:
- Kova 补货 Agent 在旺季前 2-3 周推送"春夏新款 SKU-0023 预测 20 天后断货"建议
- 抖店/批发两渠道库存统一视图，超卖时实时预警
- 月底应收应付台账一键导出

**AI 功能付费意愿点**: "如果能帮我少盘一次库存、少超卖一次，¥500/月 都合理。"

---

### Persona B: 仓管 老赵 (高频操作用户)

**背景**: 在一家电子元件经销商负责仓库，每天处理 30-80 张入库/出库单，管理 2 个仓库、约 1500 个 SKU，有条码枪。
**现状**: 用速达，入库需要电脑操作，扫码后还要手动填数量，批次管理全靠 Excel 备注。
**核心痛点**:
- 条码枪扫码后系统响应慢，扫一个 SKU 等 2-3 秒
- 批次管理混乱，快过期的货不知道优先出
- 每次盘点要关仓一天，影响出货

**Tally 解决的核心场景**:
- 扫码枪 HID 键盘模式，焦点自动捕获，扫一个入一个，响应 < 300ms
- 批次号 + 有效期字段内置，系统自动提示用先进先出批次出货
- 循环盘点模式，不需要关仓，按货区分批盘点

**AI 功能付费意愿点**: "能帮我自动提醒哪批货快到期，省得我每天翻记录。"

---

### Persona C: 业务员 小王 (移动开单用户)

**背景**: 在一家食品经销商做销售，经常在外跑客户，需要在客户现场查库存、开销售单。
**现状**: 需要打电话给仓库确认库存，纸质开单，回公司再录系统，经常出错。
**核心痛点**（来自 google.md §6）:
- 外出时无法实时查库存，开单凭感觉，超卖后很尴尬
- 手工录单出错率高，客户名称/品名常抄错
- 无法给客户现场出一张专业报价/送货单

**Tally 解决的核心场景**:
- Web 端平板/PC 出库操作；AI 助手查询"A 客户还有多少欠款"
- 自然语言开单辅助："帮我加一行：速冻水饺 30 箱"
- 打印销售单/送货单，支持打印模板

**AI 功能付费意愿点**: "能让我在客户那边直接确认库存和开单，不用打电话，就够了。"

---

### Persona D: 财务 张会计 (月末对账用户)

**背景**: 在一家小型制造商兼任财务，每月需要核对进销存流水和应收应付，配合会计师事务所做年报。
**现状**: 进销存系统和财务软件数据不同步，每月需要手工比对，经常有差异无法追溯。
**核心痛点**（来自 google.md §6）:
- 进销存数据与财税割裂，对账耗时 2-3 天
- 金税四期来临，不知道系统是否支持数电票归档
- 应收账款超期不提醒，坏账风险高

**Tally 解决的核心场景**:
- 应收应付台账实时更新，超期自动标红提醒
- 销售单与发票状态关联，金税四期 v2 集成后可直接比对
- 月末台账导出 Excel，直接给会计师

**AI 功能付费意愿点**: "只要能自动算出每个客户的应收总额和超期明细，就能省我半天。"

## 4. MVP Scope

> 完整锁定内容见 decision-lock.md §3。本节在 decision-lock 基础上细化到用户故事粒度。

### 4.1 Must-Have (MVP P0)

#### 模块 1: 账户与多租户

- **US-1.1** As a 新用户, I want 用 Zitadel OIDC 注册并创建企业账户, so that 我的数据与其他企业完全隔离。
  - 验收: 注册后自动创建 `tenant_id`，PostgreSQL RLS 策略激活，跨租户查询返回空集。
- **US-1.2** As a 企业管理员, I want 邀请团队成员并分配角色（管理员/仓管/业务/只读）, so that 权限精确控制。
  - 验收: 仓管角色无法访问财务台账页面；业务角色无法修改商品成本价。

#### 模块 2: 商品与 SKU

- **US-2.1** As a 管理员, I want 创建商品并定义多 SKU（颜色/尺码/规格组合）, so that 每个变体有独立库存和条码。
  - 验收: 一个商品可挂 N 个 SKU，每个 SKU 有独立条码、采购价、零售价、库存数量。
- **US-2.2** As a 管理员, I want 批量导入商品（Excel 模板）, so that 从旧系统迁移时不必逐条录入。
  - 验收: 提供 `.xlsx` 模板下载；导入 500 行 < 5 秒；错误行标红显示，正确行入库。
- **US-2.3** As a 仓管, I want 用条码枪扫码定位 SKU, so that 开单和盘点时不必手动搜索。
  - 验收: HID 键盘模式（无 SDK），焦点输入框自动捕获条码，扫码到定位 SKU < 300ms。
- **US-2.4** As a 管理员, I want 为商品设置安全库存阈值, so that 低库存时系统自动预警。
  - 验收: 库存量 < 安全库存 50% 显示黄色预警；< 20% 显示红色紧急预警；Dashboard 待办卡片聚合。
- **US-2.5** As a 管理员, I want 为食品类商品开启批次管理（批号 + 有效期）, so that 追溯和先进先出出货合规。
  - 验收: 批次号、有效期在入库时必填（批次管理商品）；出库时系统建议最近到期批次优先。
- **US-2.6** As a 管理员, I want 为贵重商品开启序列号管理, so that 每件商品可追溯到具体买家。
  - 验收: 序列号在出库单中一一对应；查询序列号可定位到对应销售单和买家。

#### 模块 3: 仓库与库存

- **US-3.1** As a 管理员, I want 创建多个仓库并定义各仓库的 SKU 库存量, so that 多仓库分开管理。
  - 验收: 同一 SKU 在不同仓库有独立库存；库存合计视图可选择"全仓"或"单仓"。
- **US-3.2** As a 仓管, I want 查看库存实时六状态（在手/可用/预占/在途/损坏/冻结）, so that 知道哪些库存可销售。
  - 验收: 可用库存 = 在手 - 预占；销售单确认后自动更新预占；入库确认后更新在手。
- **US-3.3** As a 管理员, I want 预留多渠道库存视图（v1 UI 展示，v2 接 API）, so that 未来多平台接入有数据基础。
  - 验收: 库存表有 `channel_id` 字段（v1 默认 `default`）；UI 预留"渠道"筛选列（v1 禁用但可见）。

#### 模块 4: 采购流程

- **US-4.1** As a 采购员, I want 创建采购单（选供应商 → 添加 SKU 明细 → 确认提交）, so that 正式发起采购流程。
  - 验收: Stepper 三步，步骤 2 行内编辑数量/单价，实时汇总；草稿自动保存（⌘S）；提交后生成 `PO-YYYYMMDD-XXX` 编号。
- **US-4.2** As a 仓管, I want 确认采购入库（扫码 + 实收数量）, so that 库存自动增加并生成入库单。
  - 验收: 入库确认后，对应 SKU `on_hand` 增加；`bill_head` 状态 → 已入库；WAC 成本自动重算。
- **US-4.3** As a 财务, I want 查看每张采购单的应付款状态（未付/部分付/已结清）, so that 控制付款节奏。
  - 验收: 采购单详情显示应付金额、已付金额、剩余应付；支持录入付款记录（金额/日期/备注）。
- **US-4.4** As a 采购员, I want 对已审核采购单进行反审/红冲, so that 纠正错误而不破坏数据完整性。
  - 验收: 反审后原单状态变为"已冲销"；系统自动生成对应红字入库单反冲库存；操作记录写入审计日志。

#### 模块 5: 销售流程

- **US-5.1** As a 业务员, I want 创建销售单（选客户 → 添加 SKU + 数量 + 价格 → 确认出库）, so that 完成一笔销售。
  - 验收: 销售单创建时校验 `available` 库存充足；超库存时阻止提交并显示实际可用量；出库确认后 `on_hand` 减少。
- **US-5.2** As a 业务员, I want 在销售单中给不同 SKU 设置折扣, so that 灵活定价不同客户。
  - 验收: 行级折扣（折扣率 or 折扣金额）和单据级整体折扣均支持；折后金额实时汇总。
- **US-5.3** As a 财务, I want 查看每张销售单的应收款状态, so that 催款有据可查。
  - 验收: 应收台账按客户汇总；支持录入收款记录；超期应收自动标红（可配置超期天数）。
- **US-5.4** As a 财务, I want 对销售单进行红冲（销售退货）, so that 处理客户退货时账面正确。
  - 验收: 红冲生成退货入库单，库存恢复；对应应收自动抵减；审计日志记录操作人。

#### 模块 6: 调拨与盘点

- **US-6.1** As a 仓管, I want 发起跨仓库调拨单, so that A 仓库存可转移至 B 仓。
  - 验收: 调拨单确认后 A 仓 `on_hand` 减少，B 仓 `on_hand` 增加；调拨过程中显示"在途"状态。
- **US-6.2** As a 仓管, I want 发起整仓盘点, so that 实盘数量与系统账面核对并处理差异。
  - 验收: 盘点单显示账面库存；实盘数量行内输入（Tab 跳行，Enter 确认）；差异自动计算；盘点差异单审核后库存调平。
- **US-6.3** As a 仓管, I want 支持循环盘点（按商品分类/货架分批盘）, so that 不必关仓一次性盘完。
  - 验收: 可选择"部分 SKU 盘点"；盘点任务状态跟踪（进行中/待审核/已完成）；进度条显示当前盘点覆盖率。

#### 模块 7: 财务台账

- **US-7.1** As a 财务, I want 查看应收账款台账（按客户/按时间/按状态）, so that 掌握回款情况。
  - 验收: 台账按客户分组，显示总应收/已收/未收；支持 Excel 导出；超期金额高亮。
- **US-7.2** As a 财务, I want 查看应付账款台账（按供应商）, so that 控制现金流。
  - 验收: 同应收台账对称；支持多次付款拆分录入；付款完成后自动结清。
- **US-7.3** As a 财务, I want 查看资金账户余额和收支流水, so that 对现金账户有基本了解。
  - 验收: 支持多个资金账户（银行账户/支付宝/微信）；每笔收付款关联到对应应收/应付单据。

> **Scope 说明**: v1 财务模块是"简化应收应付台账"，不做总账/科目/凭证，不对接用友/金蝶。

#### 模块 8: 报表

- **US-8.1** As a 老板, I want 查看仪表盘（月销售额/毛利率/库存价值/欠款总额）, so that 每天 5 秒看懂业务状态。
  - 验收: 四个 KPI 卡，每个含当前值/环比/Sparkline；Dashboard 首屏加载 < 1.5s（LCP 指标）。
- **US-8.2** As a 老板, I want 查看库存周转率报表（按商品/按类别）, so that 找出资金占压最严重的品类。
  - 验收: 支持自定义时间范围；周转率 = 出库成本 / 平均库存价值；支持 Excel 导出。
- **US-8.3** As a 老板, I want 查看 ABC 分析（按销量/销售额分三档）, so that 聚焦高价值 SKU。
  - 验收: A 类（前 20% SKU 贡献 80% 销售额）、B 类、C 类清晰标注；支持按时间段重算。
- **US-8.4** As a 老板, I want 查看滞销预警报表（超过 N 天未动库存的 SKU）, so that 主动清库减损。
  - 验收: 可配置滞销天数阈值（默认 30 天）；列表显示当前库存量/库存金额/最后出库时间。

#### 模块 9: AI 助手 (Hub + Kova)

- **US-9.1** As a 任何用户, I want 用 ⌘K 唤起自然语言查询 + AI Drawer 获取答案, so that 不用记报表路径也能得到数据洞察。
  - 验收: ⌘K 打开 Command Palette；输入"本月滞销最严重的 10 个 SKU"→ AI Drawer 流式返回表格+分析；Hub API P95 响应 < 3s（含流式首字节）。
- **US-9.2** As a 老板, I want Kova 补货 Agent 推送补货建议（带预测依据）, so that 不用自己分析销量就能决策。
  - 验收: Kova Agent 基于历史销量 + 安全库存计算推荐补货量；建议以"待办卡片"形式出现在 Dashboard；用户可一键"采纳"（跳转到预填采购单）或"忽略"；v1 仅建议，不自动提交。
- **US-9.3** As a 老板, I want 收到滞销预警推送（Dashboard 卡片 + 系统通知）, so that 及时处置积压。
  - 验收: Kova Agent 每日运行一次；滞销 SKU（超 30 天未动）推送 Dashboard 卡片；AI 给出处置建议（降价/调拨/退供）供用户参考，不自动执行。

### 4.2 Defer 到 v2 (MVP 不做)

| 功能 | 原因 | 预计触发条件 |
|------|------|------------|
| 金税四期 ISV 集成（航信/百望云） | 选型 + 接入周期 3 个月；v1 留接口 | 付费客户明确提出合规需求 |
| 抖店/拼多多/淘宝 OAuth 库存同步 | 平台审核周期长；v1 预留 `channel_id` 模型 | Stage 阶段客户验证需求 |
| FIFO 成本算法 | v1 WAC 满足 80% 客户；食品/医药客户驱动 FIFO | 5+ 客户明确要求 |
| 多币种 | v1 CNY only；出海客户驱动 | 进入东南亚市场时 |
| 货位（bin/shelf）管理 | 增加仓库复杂度；大多数中小客户用不到 | 客户 SKU > 2000 且有货位需求 |
| Kova Agent 自动执行（无需审核） | 用户信任需先建立；v2 加可撤销自动执行 | v1 采纳率稳定 > 60% 后 |
| 用友/金蝶财务对接 | 大客户需求；v1 台账满足小客户 | 客户有明确对接需求 |
| 供应商询价 Agent | 需要供应商端配合；工期较长 | Kova 多 Agent 编排成熟后 |
| TanStack Virtual 虚拟滚动 | v1 目标客群 SKU < 3000，按需激活 | 单租户 SKU 实测超 2000 时 |
| pgvector 商品 embedding | 字段已预留；v1 不强制写入 | AI 相似商品推荐功能启动时 |

### Journey 3: 每日开单（销售单 → 出库 → 库存扣减）

**角色**: 业务员小王 | **触发**: 客户下单

1. ⌘K → 输入"新建销售单"或点击左侧导航"销售 → 新建"
2. 步骤 1: 选择客户（搜索或下拉，支持快速创建新客户）
3. 步骤 2: 添加商品明细
   - 搜索框支持商品名/SKU 编码/助记码/条码扫码
   - 扫码枪扫码 → 自动定位 SKU → 光标跳至数量列（见 ux-benchmarks §4.5 扫码逻辑）
   - 数量/单价行内编辑，实时汇总金额
   - 系统实时显示各 SKU 可用库存；超库存时行内警告（不阻断录入，允许录入后提示）
4. 步骤 3: 确认出库（选择仓库、付款方式、备注）
5. 提交 → 乐观更新 → Toast "销售单 SO-20260423-001 已创建" + [撤销] 3s
6. 出库确认（仓管操作）→ 库存 `on_hand` 自动扣减

**验收标准**:
- Given 可用库存 50 件，When 销售单提交 60 件，Then 提交时显示警告"可用库存不足（50 件）"，阻止出库确认但允许保存草稿。
- Given 出库单已确认，When 查看库存，Then 对应 SKU `on_hand` 精确减少出库数量，WAC 成本不变（出库不影响成本）。

### Journey 4: 周/月盘点（扫码盘点 + 差异处理）

**角色**: 仓管老赵 | **触发**: 月末盘点

1. 左侧导航"库存 → 盘点 → 新建盘点" → 选择"整仓盘点"或"循环盘点（按类别）"
2. 系统生成盘点任务：拉取当前账面库存作为"应盘数量"
3. 盘点界面（见 ux-benchmarks §4.5 盘点页设计）:
   - 顶部进度条：已盘 / 总数
   - 扫码枪扫码 → 聚焦对应 SKU 行，光标在"实盘数量"列
   - 键盘 Tab 跳下一行，Enter 确认当前行
   - 差异列实时计算（红色高亮差异 > 5% 的行）
4. 全部盘完 → 提交盘点差异报告 → 管理员审核
5. 审核通过 → 系统自动调平库存（正差生成盘盈入库单，负差生成盘亏出库单）

**验收标准**:
- Given 账面库存 100 件，When 实盘输入 95，Then 差异列显示 `-5 (-5%)`，红色高亮；提交后待审核状态。
- Given 盘点差异审核通过，When 查看库存，Then 库存调整为 95，操作日志记录"盘点调平 -5"。

### Journey 5: AI 自然语言查询（⌘K + Drawer 流式回答）

**角色**: 老板娘李青 | **触发**: 月末想了解库存情况

1. 任意页面按 ⌘K → Command Palette 弹出（见 ux-benchmarks §4.8）
2. 输入"本月滞销最严重的 10 个 SKU 并给出处置建议"
3. 系统识别为 AI 查询 → 自动打开右侧 AI Drawer（见 ux-benchmarks P12 + §4.7）
4. Hub API 流式返回：
   - 首先返回 Markdown 表格（SKU 名称 / 库存量 / 库存金额 / 最后出库日期）
   - 然后返回分析段落（哪些可清货、哪些可退供、哪些季节性可等）
   - 嵌入操作按钮：[查看 ABC 报表] [创建调拨单]
5. 用户点击[查看 ABC 报表] → 跳转报表页并高亮对应 SKU

**支持的查询类型** (v1 范围):
- 库存查询："XX 商品现在有多少库存？" / "哪个仓库的台灯最多？"
- 销售分析："本月销售额最高的 10 个 SKU" / "和上月比较销量变化"
- 应收应付："A 客户还欠多少钱？" / "哪些供应商的款项超期了？"
- 预警汇总："现在有哪些低库存预警？" / "最近 30 天有哪些滞销商品？"
- 补货建议："台灯 A 款还能卖几天？需要补多少货？"
- 报表生成："帮我生成本月销售报表"（返回可下载链接）

**不支持** (v1 边界，见决策点 §2):
- 直接执行操作（"帮我创建采购单"→ 仅跳转到预填表单，需用户确认提交）
- 财务凭证生成、开票操作

**验收标准**:
- Given 用户输入库存查询，When Hub API 返回，Then Drawer 首字节响应 < 1.5s，流式完整输出 < 5s。
- Given AI 回答包含操作建议，When 用户点击嵌入按钮，Then 跳转到对应页面（表单预填或报表定位）。

### Journey 7: 滞销预警处理（Kova Agent 推送）

**角色**: 老板娘李青 | **触发**: 每日 09:00 Kova Agent 运行

1. Dashboard 待办卡片区出现："[AI] 发现 5 个 SKU 滞销超 30 天，库存价值 ¥12,340"
2. 点击卡片 → 展开详情（滞销 SKU 列表 + Kova 建议）:
   - "台灯 A款 (TD-A001)：库存 80 件，最后出库 45 天前。建议：降价 15% 促销 或 调拨至 B 仓"
   - "充电宝 C款 (CB-003)：库存 200 件，3 个月未动。建议：退供应商或打包处置"
3. 用户对每条建议选择操作:

   - **忽略** → 记录用户决策，下次不再推送此条（7 天内）
   - **暂缓** → 3 天后再次推送
4. 操作完成 → 卡片变为"已处理"状态，归档到历史

**验收标准**:
- Given Kova Agent 检测到 SKU 滞销，When 用户采纳调拨建议，Then 跳转调拨单页面，目标商品和数量已预填，用户只需选择目标仓库并确认。

### Journey 8: 多仓库调拨

**角色**: 仓管老赵 | **触发**: A 仓库存过剩，B 仓缺货

1. 左侧导航"库存 → 调拨 → 新建调拨单"
2. 选择：源仓库（A 仓）→ 目标仓库（B 仓）→ 添加 SKU + 数量
3. 系统验证：A 仓可用库存是否充足
4. 提交 → 调拨在途：A 仓 `available` 减少，状态为"在途"
5. B 仓确认收货 → B 仓 `on_hand` 增加，调拨完成
6. 支持部分收货（B 仓只收到 80%，剩余 20% 继续在途）

**验收标准**:
- Given A 仓可用 100 件，When 提交调拨 60 件至 B 仓，Then A 仓 `available` 变为 40，显示"在途 60"；B 仓确认后 `on_hand` + 60，A 仓在途归零。

### 6.1 商品模块

| 功能 | 优先级 | 验收标准 |

| 商品主信息 (名称/分类/品牌/型号/规格/助记码) | P0 | 必填字段校验；助记码自动生成汉字拼音首字母 |

### 7.3 安全与合规

- **多租户隔离**: PostgreSQL Row-Level Security，`app.tenant_id` 每请求通过 GORM 中间件注入；RLS 策略验证写入测试覆盖（见 Lurus CLAUDE.md TDD 章节）。
- **认证授权**: Zitadel OIDC，JWT 验证；RBAC 四角色（管理员/仓管/业务/只读）；每个 API 路由强制鉴权。
- **操作审计日志**: 所有写操作（创建/修改/删除/红冲/审核）写入 `audit_log` 表（记录 who/what/when/result）；不可被普通用户删除。
- **数据本地化**: 所有业务数据存储在 `lurus-pg-rw`（国内服务器）；AI 调用通过 Hub 代理，不直接暴露客户数据给外部 LLM。
- **传输加密**: 全程 HTTPS/TLS 1.3；API 和前端均走 Traefik Ingress，TLS 证书自动续签。
- **输入校验**: 所有外部输入在 Handler 层 Zod 校验（前端）+ Gin binding 校验（后端）；金额字段使用 `NUMERIC(18,4)` 避免浮点误差。

## 9. AI 差异化策略（Lurus 独有）

### 9.1 Hub 自然语言查询 (v1 Day 1)

**定位**: 替代"用户记报表路径"的学习成本，降低从管家婆/速达迁移的操作摩擦。

**支持查询类型** (v1，见 Journey 5 完整清单):
- 库存状态查询（指定商品/仓库/批次）
- 销售数据分析（时间段对比、Top N）
- 应收应付状态查询
- 低库存/滞销预警汇总
- 补货需求估算（基于历史销量推算天数）
- 报表生成（AI 生成并提供下载链接）

**技术路径**: Hub API → DeepSeek/Claude → 结构化 SQL 查询结果格式化 → Markdown 输出 → Drawer 流式渲染。

**v1 边界**: AI **不直接执行任何写操作**。AI 可以建议"帮我查一下 A 客户欠款"并返回数据，可以建议"应该补货 100 件"并在 Sheet 预填参数，但用户必须点击"确认提交"才能真正执行。

### 9.2 Kova 补货 Agent (v1 建议级)

**定位**: 国内首个 PSI Agent 级补货决策——不是规则引擎，是真正基于 Kova 推理引擎的 Agent。

**v1 工作方式**:
1. Kova Agent 每日 09:00 运行（Cron 触发）
2. 读取历史 90 天销量数据 + 当前库存 + 安全库存阈值
3. 计算每个 SKU 的预计断货时间（基于日均销量 + 趋势）
4. 生成补货建议列表（SKU / 建议采购量 / 建议供应商 / 理由摘要）
5. 建议写入 `agent_recommendations` 表，推送到 Dashboard 待办卡片
6. 用户审核：采纳（跳转预填采购单）/ 忽略（7 天内不重推）/ 暂缓（3 天后重推）

**v2 规划**（需先建立用户信任）:
- 用户可设置"自动执行阈值"（如"建议采购量 < ¥500 自动创建草稿采购单"）
- Kova 执行采购单草稿，等待用户审批（可撤销）
- Kova 多 Agent：自动询价 → 汇总报价 → 建议最优供应商

**v3 规划**:
- Kova 长任务 Agent：端到端从需求信号到 PO 发出（全自动 + 异常人工干预）

### 9.3 Memorus RAG 上下文 (v1 可选集成)

**定位**: 历史销量和客户偏好上下文，提升 AI 预测精度。

**v1 实施**:
- Memorus 存储：每租户的历史销量序列（SKU + 日期 + 销量）、客户下单偏好
- Hub 查询时带入 Memorus 上下文 → 更准确的季节性预测
- embedding 字段已在 `product` 表预留（`vector(1536)`），但 v1 不强制写入

### 9.4 多渠道库存智能分配 (v2 预留)

数据模型已在 v1 预留 `channel_id` 字段。v2 接入抖店/拼多多 OAuth 后，Kova Agent 可根据各平台历史转化率建议库存分配比例。

### 9.5 金税四期 AI 合规巡检 (v2 预留)

v1 留接口（`tax_integration` 适配层空实现）。v2 选型 ISV 对接（航信/百望云/诺诺），由 AI 对数电票 XML 做四流一致性校验，异常推送 Dashboard 待办卡片。

### 9.6 计费模式

- **基础订阅**: 走 `2l-svc-platform` 订阅体系（月付/年付）
- **AI 调用按量计费**: 每次 Hub LLM 调用、Kova Agent 执行均计量；超出套餐额度扣钱包
- **权益分层** (TBD by first customers): 免费试用 14 天；基础版（含 N 次/月 AI 调用）；专业版（更多 AI + 优先响应）
