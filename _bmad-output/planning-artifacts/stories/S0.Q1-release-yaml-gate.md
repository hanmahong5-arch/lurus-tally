## Story

As a release pipeline maintainer,
I want `image-backend` + `image-web` jobs to be gated by CI lint / test / frontend success,
so that a red `errcheck` / failing unit test never produces a published `ghcr.io/...:main-<sha>` image.

## Context — `needs:` 跨 workflow 不工作

GitHub Actions `needs:` **只在同一 workflow file 内有效**。`release.yaml` 与 `ci.yaml` 是两个独立 workflow，**不能** 直接 `needs: [backend-lint, backend-test, frontend]`。3 种可选实现：

| 方案 | 代价 | 推荐 |
|---|---|---|
| A. 把 image-* 两个 job **挪进 ci.yaml** | 1 个 workflow 变长，但 `needs:` 原生 | ✅ 推荐 |
| B. release.yaml 用 `workflow_run: workflows: [CI], types: [completed]` 触发 | trigger 改成事件驱动，丢 `push` 即时性 | ⚠️ 备选 |
| C. release.yaml 内部跑 `gh run view --workflow=ci.yaml` poll | 脆 + 速率限制 | ❌ 不推荐 |

**默认走 A。** 详情见 Dev Notes。

## Acceptance Criteria

1. 在 main 分支故意提交一条 `errcheck` 违规（如 `_ = json.Marshal(...)` 改为 `json.Marshal(...)` 不接收 err），push 后：
   - `backend-lint` job FAIL
   - `image-backend` + `image-web` job 状态为 **`skipped`**（不是 `success` 也不是 `failure`）
   - `ghcr.io/hanmahong5-arch/lurus-tally-{backend,web}:main-<sha7>` **不**出现新 tag
2. Revert 该违规后下一次 push，image-* 两个 job 正常 `success` 并 push 镜像。
3. 文档更新 `doc/coord/changelog.md` 说明 gate 已上线 + 影响。
4. 不能 `--no-verify` 跳 hook；遇 hook 红 → 修根因。
