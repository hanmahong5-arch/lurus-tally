## Story

As 产品负责人在 Sprint 3 阶段一 gate review，
I want 一份可重复的相关性 eval（3 个真实页面 × 5 问 × 人工打分 ≥ 4/5）数据，
so that F1.4 验收"3 个真实页面回答相关性 ≥ 4/5（n=15 内测）"不是拍脑袋，是有据可查的 evidence — 不达标就明确告诉团队"Drawer v0.5 不能 ship, 加 sprint 修"。

## Acceptance Criteria

1. Eval 数据集 `_research/eval/drawer-v0.5-q15.yaml`：15 个 (page_context, question, reference_answer) 三元组，覆盖 product / sales_bill / dashboard 三 entity_kind 各 5 个；question 来自团队真实使用 / 客户访谈预设而非 LLM 生成。
2. Eval 脚本 `_research/eval/run-drawer-eval.sh`：
   - 起 local backend（`go run ./cmd/server`）+ 用 seed tenant
   - 逐条调 `/api/v1/ai/chat single_shot=true` 收集 SSE 输出
   - 把 (question, answer, reference) 写到 `_research/eval/results-YYYYMMDD-HHmm.md`
3. 评分 rubric `_research/eval/rubric.md`：5 分制，含 4 个维度（相关性 / 准确性 / 简洁度 / actionable），单分聚合用平均；2 个评分人独立打分（创始人 + 内测客户 1 名），inter-rater 一致率 ≥ 70%（差距 ≤ 1 分占比）。
4. **必须真跑 LLM**——禁止用 mock / regex 评分（见 ~/.claude/CLAUDE.md §4.1 第 ③ 条 "测量与被测不可同源"）。
5. Eval 输出至少 12/15 分 ≥ 4.0 = 通过；< 12/15 → 写 `_research/eval/failure-analysis.md`，列 cluster（context 摘要不全 / token 截断 / prompt 措辞模糊）+ 改进项 → 决策是否阻塞 sprint 3 gate。
6. 报告 commit message **必须**带 measured before/after（CLAUDE.md §4.1 第 ⑤ 条），格式 `eval(drawer-v0.5): n=15 mean=4.X median=Y.Z passed=12/15`。

## Dev Notes

- **死守 honesty**：reference_answer 自己写（基于 seed 数据），不让 LLM 写然后 LLM 再被这个 reference 打分（self-validating loop，§4.1 ③）
- 如果某天跑完是"15/15 全 5 分"——立刻怀疑：是不是 question 太简单 / reference 抄了 model output；按 §4.1 ① "完美数字 = 反射性怀疑"内省
- 不达标不要"再加 5 个 prompt 凑过"——按 failure-analysis 找根因（context 摘要 / prompt 措辞 / 模型容量）然后开新 story 修
- 这个 eval 还会被 F2.1（补货 agent）复用模板 — 写脚本时考虑可扩展（参数化 endpoint + dataset 路径）
