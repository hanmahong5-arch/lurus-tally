## Story

As a product analyst,
I want 5 new typed telemetry events (`palette_invocation`、`ai_drawer_open`、`plan_accept_rate`、`onboarding_first_po_exported`、`cmd_z_used`) 写到 NATS `PSI_TELEMETRY.web.*`,
so that Sprint 1-12 的"⌘K DAU ≥60%"、"AI Drawer 相关性"、"onboarding 30 min"、"Cmd+Z 月 5 次"等量化目标能从前端真实埋点 aggregator 出来。

## Acceptance Criteria

1. `web/lib/telemetry.ts:8` `TelemetryEvent` union 从 2 扩到 7：保留 `draft_restore` / `undo_used`，新增 `palette_invocation` / `ai_drawer_open` / `plan_accept_rate` / `onboarding_first_po_exported` / `cmd_z_used`。
2. 每个 event 的 metadata 字段 schema 文档化在 `web/lib/telemetry.ts` JSDoc：
   - `palette_invocation`: `{latency_ms, query_chars, action_picked: 'navigate|query|execute|none'}`
   - `ai_drawer_open`: `{page_context, trigger: 'shortcut|button|deeplink'}`
   - `plan_accept_rate`: `{plan_id, kind: 'replenishment|movement|...', accepted: '1|0'}`
   - `onboarding_first_po_exported`: `{tenant_age_minutes, export_format: 'csv|pdf'}`
   - `cmd_z_used`: `{entity_type, undo_latency_ms}`
3. `web/app/api/otel-events/route.ts` 接收 POST 后，**除了**写 OTel（如已有）**额外**用 NATS publisher 发到 subject `PSI_TELEMETRY.web.<event_name>`；payload = `{tenant_id, user_id, event, metadata, timestamp}` JSON。
4. NATS publisher 调用后端新 endpoint `POST /internal/v1/telemetry/web` 转发（前端不直连 NATS）；endpoint 走 same-origin cookie auth；后端用现有 `internal/adapter/nats/publisher_typed.go` 加新 typed method `PublishWebTelemetry`。
5. Playwright e2e test：5 个新 event 全部触发一次，断言 NATS subject `PSI_TELEMETRY.web.>` consumer 收到 5 条对应 payload。
6. 不破坏现有 `draft_restore` / `undo_used` 流；现有调用点（如 `restoreSnapshot` 等）回归测试通过。

## Dev Notes

- **NATS subject vs stream**：现 `PSI_EVENTS` stream 大概率只接 `PSI_EVENTS.>`。新增 `PSI_TELEMETRY.>` 需要：1) 在 stream 配置加 `Subjects: ["PSI_EVENTS.>", "PSI_TELEMETRY.>"]` 或 2) 新建独立 `PSI_TELEMETRY` stream（推荐 2，避免业务/遥测混 retention）。Dev 实施前 `nats stream info PSI_EVENTS` 确认。
- **PII 守卫**：metadata 字段 allow-list 严格 — 不接受 `user_email` / `note` 等自由文本（避免日志泄漏）。
- 前端 `trackEvent` 仍 fire-and-forget；NATS publish 失败不能 throw 给 UI。
- backend handler 失败时返 204（不阻塞 UI）+ structured log `WARN` 让 ops 排查。
- 不要在前端直接连 NATS — websocket NATS 暴露 server 不在本 sprint 范围。

### Decisions Made

- **NATS stream subject 扩展未上线**: 现 `PSI_EVENTS` stream 只接 `PSI_EVENTS.>`。`PSI_TELEMETRY.web.*` 在代码层 publish 但 stream config 未加 — 真发布会被 server reject。落地选择两条:
  1. 在 deploy/k8s 给 NATS server 加 `subjects: ["PSI_EVENTS.>", "PSI_TELEMETRY.>"]`（推荐：单 stream，30d retention）
  2. 新建独立 `PSI_TELEMETRY` stream（独立 retention 调优）
  S0.Q4 完成 PG backup 后由 devops 在 STAGE 部署同时加 stream config，作为 S0 sprint exit 一部分
- **Playwright e2e 未跑**: 需要 NATS server + Tally backend 同时 running，目前 dev 环境不齐；e2e 落地放 S1（与 STAGE 客户使用一起 verify）
- **AC 偏差**: plan AC5 要 Playwright assert 5 个 event 到 NATS；本 sprint 落 7 个 unit test + 12 个 vitest 替代，待 e2e 补
- **PII allow-list 严格**: handler 不解析 metadata schema（保持 `map[string]any`），由 FE TypeScript 类型保证。后续若有滥用可在 handler 加 field-level allow-list 二次过滤
