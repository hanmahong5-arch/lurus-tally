## Story

As an operator,
I want daily `pg_dump` of Tally schema → MinIO with 14d retention + a one-time restore drill,
so that 部署成熟度 5.0 → 7+ 子分项里的"无 PG backup"打钩，并且 R6 STAGE 任何一次磁盘 / 数据损坏可在 1h 内恢复。

## Acceptance Criteria

1. 新建 `deploy/k8s/base/cronjob-pgbackup.yaml`：daily 02:00 UTC（北京时间 10:00，避开高峰）调度，`pg_dump --schema=tally --format=custom` → `s3://tally-backup/<YYYY-MM-DD>.dump`。
2. CronJob image 用官方 `postgres:16-alpine` + `mc` (MinIO client) 二阶段：先 `pg_dump` 到 emptyDir，再 `mc cp` 到 MinIO；不在容器里堆装 awscli。
3. 14 天保留 — 在 `mc cp` 后跑 `mc rm --recursive --older-than 14d`；用 ETL 守护手势单文件失败不影响后续。
4. CronJob 通过 K8s secret 读 `PG_PASSWORD` + `MINIO_ACCESS_KEY` + `MINIO_SECRET_KEY`；不允许任何 plaintext 凭证写 manifest。
5. `STAGE_RUNBOOK.md`（新建或扩展）加 §Restore Drill 章节，详细命令：
   - `kubectl create job --from=cronjob/tally-pgbackup pgbackup-manual-$(date +%s)` 触发一次
   - `mc cp s3://tally-backup/<date>.dump /tmp/`
   - `psql -c "CREATE SCHEMA tally_restore_test;" && pg_restore --schema=tally --no-owner -d $PG_URL --schema-rename=tally:tally_restore_test /tmp/<date>.dump`
   - `psql -c "DROP SCHEMA tally_restore_test CASCADE;"` 清场
6. **本 sprint 实际跑过一次 restore drill**：dump 大小 > 1 MB（确认非空）；restore 后 `SELECT count(*) FROM tally_restore_test.product` ≈ stage 实际行数（±5%）；drill log 贴在 story Dev Agent Record。
7. CronJob 部署到 R6 STAGE（不部署 R1，PROD 无 ArgoCD 接管，参考 ADR-0006 走人工 apply）。

## Dev Notes

- **不上 WAL archiving** — 增量备份对 Tally 当前数据量过重；daily full dump + 14d 保留对前 10 客户够用。V2 客户超 100 再评估。
- **MinIO vs S3** — STAGE 走 R6 internal MinIO；PROD 留待 R2 planning 决定。
- Cron 02:00 UTC 是磁盘 SSD HARD RULE 友好窗口（写 dump 到 `/data`）；emptyDir 不要超 1Gi（限 resources）。
- 14d 保留是合规底线；客户付费后调到 30d。
- 不要把 dump 写 `/var/lib/rancher`（CN 项目 CLAUDE.md R6 磁盘 HARD RULE）。
- restore drill log 贴到 story 完成后 `Dev Agent Record`，是 exit gate 唯一硬证据（不允许"我跑过了"无 log）。

### Decisions Made

- **不上 WAL archiving**：每日 full dump + 14d 对前 10 客户够；V2 客户超 100 再评估增量。
- **CronJob image 用 postgres:16-alpine + 二阶段下载 mc**（节省 image 维护）；postgres version 跟当前 STAGE 一致避免 dump 格式漂移。
- **dump <1MB abort 是真测**：tally schema 至少有 currency/migration_locks 等表，绝不为空。<1MB 一定是 pg_dump 出错（auth fail / schema 名错），不是数据少。
