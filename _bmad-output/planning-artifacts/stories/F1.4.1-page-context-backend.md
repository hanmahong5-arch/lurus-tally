# Story F1.4.1: AI Drawer v0.5 — Page-Context Backend + Single-Shot Answer

Status: ready-for-dev

**Epic:** E17 (Kova Agent) 前置 / E18 协同
**Sprint:** 2-3 (阶段一)
**Feature:** F1.4 AI Drawer v0.5 (80h total; this story = 24h)
**Owner:** tally backend
**Repo:** `2b-svc-psi`
**Estimate:** 24h
**Blockers:** 无（复用现有 `/api/v1/ai/chat` SSE + orchestrator）

## Story

As 用户在某个 SKU 详情页打开 AI Drawer，
I want 后端拿到我当前所在页面的 entity 上下文（`product_id=sku-12345`），
so that 我问"这个 SKU 周转怎么样" 时 AI 直接回答**这个**而不是反问"你说的是哪个 SKU"，且首字回应 < 1.5s。

## Acceptance Criteria

1. `POST /api/v1/ai/chat` 请求体扩字段 `page_context: {route, entity_kind?, entity_id?, extra?:{}}`；现有不带 context 调用方仍兼容（字段 optional）。
2. orchestrator 在 system prompt 顶部注入页面上下文摘要，e.g.
   `当前页面: /products/sku-12345 (SKU "USB-C 数据线 1m")；最近 30 天销量 240, 在库 18, 安全线 30`。摘要由后端**查 1 次 DB**生成（不让 LLM 工具调用拿，省 token + 减延迟）。
3. 上下文摘要走 Redis cache（key `page_ctx:{tenant}:{kind}:{id}`, TTL 60s, **必须带 tenant_id 防跨租户泄漏**）。
4. 新模式 `single_shot=true`：跳过 multi-turn tool-calling、直接一次性 LLM 调用返答案；max_tokens=300；适合 Drawer v0.5"一句话回答"场景。
5. 首字 latency P95 ≤ 1500ms（SSE first chunk arrival, measured server-side with `span.AddEvent("first_chunk")`）；不达标在 PR 描述里贴 trace 截图说明原因。
6. audit_log 写入 `source='drawer'` + `page_route` 字段；扩 audit_log schema 加 `page_route TEXT` 列（migration）。

## Dev Notes

- BuildContextSummary 必须**纯函数**层（接 repo 接口，不直 GORM）— 测试可 mock；摘要 ≤ 400 token，否则浪费 LLM 上下文
- single_shot 不接 tool-calling = 故意限制；F2.1 才让 Drawer 进多步 agent 模式
- 1500ms 首字红线 = 用户哲学原则 ① "AI 不是卖点,自动化才是"——慢的"AI"=没人用
- 务必跑 trace 而不是肉眼测；P95 用 ≥ 30 次样本
